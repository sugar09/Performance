# TBChat Cloud - Image Upload Performance Test

## First-time setup (after clone)

### 1. Install tools
- Python 3.x: https://www.python.org/downloads/
- Node.js: https://nodejs.org/
- k6: https://k6.io/docs/get-started/installation/
- Pillow: `pip install Pillow pillow-avif-plugin`

### 2. Get files from Dev (not included in repo)
Contact Dev to get the following files and place them in the correct location:

| File | Location |
|------|---------|
| `buildFileTool.exe` | `genfile/upload/` |
| `uploadTool.exe` | `genfile/upload/` |
| `wallet.bin` | `genfile/upload/conf/` |

### 3. Create required folders
```powershell
mkdir D:\TBChatPerformance\assets\images
mkdir D:\TBChatPerformance\data
mkdir D:\TBChatPerformance\reports
```

## Test run process

### Step 1 — Generate test images
```powershell
cd D:\TBChatPerformance
python scripts/generate_images.py --count 1000
```
Generates 1000 unique images into `assets/images/` (auto-deletes old images before generating new ones), randomly across 5 formats: `jpg, jpeg, png, webp, avif`.

Options:
- `--count <n>` — number of images to generate (default 1000)
- `--output-dir <path>` — output folder (default `assets/images`)
- `--formats <list>` — comma-separated list of formats (default `jpg,jpeg,png,webp,avif`)

```powershell
python scripts/generate_images.py --count 1000 --formats jpg,png
```

---

### Step 2 — Process images with buildFileTool
```powershell
cd D:\TBChatPerformance\genfile\upload
.\buildFileTool.exe -uploadDir="D:\TBChatPerformance\assets\images"
cd D:\TBChatPerformance
```
The tool generates:
- `index/<unique>` — JSON body for POST init
- `upload/http_<unique>_` — binary slice for PUT to server

> **Note:** You must `cd` into the `genfile/upload` folder before running the tool.

---

### Step 3 — Build manifest
```powershell
node scripts/prepare-manifest.js
```
Generates `data/upload-manifest-image.json` for k6 to read.

> Every time `buildFileTool.exe` is re-run, this step must be re-run too.
> The script auto-detects the project root, so it can run from anywhere (no need to `cd` into a specific folder).

---

### Step 4 — Run the test

```powershell
k6 run k6-tests/Cloud/upload-image/<TC-name>.js
```

**Select the test server** via the `ENV` variable (defined in `k6-tests/Cloud/config/enviroment.js`):

```powershell
k6 run -e ENV=qa k6-tests/Cloud/upload-image/TC01-baseline.js   # default if -e ENV is not passed
k6 run -e ENV=dev k6-tests/Cloud/upload-image/TC01-baseline.js  # requires baseUrl for "dev" to be set in enviroment.js first
```

## Video test process

Videos are much larger than images (measured throughput to node1: ~1.5 MB/s, so a 20MB
video takes ~14s to PUT). `buildFileTool.exe` rebuilds `genfile/upload/index` and
`genfile/upload/upload` from scratch on every run — it does **not** append — so images
and videos cannot coexist there at the same time. Each dataset needs its own manifest
file (`data/upload-manifest-image.json` for images vs `data/upload-manifest-video.json` for
videos) so you don't have to rebuild one to run the other, but the underlying `genfile`
folders always reflect whichever dataset was built most recently.

### Step 1 — Generate test videos
```powershell
cd D:\TBChatPerformance
python scripts/generate_videos.py --count 10 --min-size-mb 5 --max-size-mb 20
```
Generates random-binary video-sized files into `assets/videos/` across 5 container
formats: `mp4, mov, avi, mkv, webm`. The server does not validate actual video/codec
content, so synthetic binary data is sufficient for load testing.

Options:
- `--count <n>` — number of videos to generate (default 10)
- `--min-size-mb <n>` / `--max-size-mb <n>` — size range per video, MB (default 5-20)
- `--formats <list>` — comma-separated list of formats (default `mp4,mov,avi,mkv,webm`)
- `--output-dir <path>` — output folder (default `assets/videos`)

### Step 2 — Process videos with buildFileTool
```powershell
cd D:\TBChatPerformance\genfile\upload
.\buildFileTool.exe -uploadDir="D:\TBChatPerformance\assets\videos"
cd D:\TBChatPerformance
```

### Step 3 — Build the video manifest (separate file, via MANIFEST_OUT)
```powershell
$env:MANIFEST_OUT = "D:\TBChatPerformance\data\upload-manifest-video.json"
node scripts/prepare-manifest.js
```

### Step 4 — Run the video test
```powershell
k6 run k6-tests/Cloud/upload-video/<TC-name>.js
```

`core-video.js` uses a 180s per-request timeout and much looser latency thresholds
than `core-image.js` (images), since a single large PUT can legitimately take tens of seconds.

## Mixed test process (images and videos uploaded at the same time)

Since `buildFileTool.exe` always rebuilds `genfile/upload/index` and `genfile/upload/upload`
from scratch, images and videos built separately can never coexist there — one always
overwrites the other. To test both being uploaded concurrently in the *same* run, generate
both into the *same* folder first, then run `buildFileTool.exe` **once** against that
combined folder.

### Step 1 — Generate test images and videos into the same folder
```powershell
cd D:\TBChatPerformance
python scripts/generate_images.py --count 50 --output-dir assets/mixed
python scripts/generate_videos.py --count 50 --min-size-mb 5 --max-size-mb 20 --output-dir assets/mixed
```
This produces 6 files total, enough for `TC01-concurrent-6-users.js`. To run
`TC02-concurrent-100-users.js` instead, use `--count 50` for both (100 total).

> Both generator scripts only clean up their *own* filename prefix (`test-img-*` /
> `test-video-*`), so running them one after another into the same `--output-dir` does
> not delete each other's files.

### Step 2 — Process the combined folder with buildFileTool
```powershell
cd D:\TBChatPerformance\genfile\upload
.\buildFileTool.exe -uploadDir="D:\TBChatPerformance\assets\mixed"
cd D:\TBChatPerformance
```
It doesn't care about file type, so it happily builds chunks for images and videos
together in a single pass.

### Step 3 — Build the mixed manifest (separate file, via MANIFEST_OUT)
```powershell
$env:MANIFEST_OUT = "D:\TBChatPerformance\data\upload-manifest-mixed.json"
node scripts/prepare-manifest.js
```

### Step 4 — Run the mixed test
```powershell
k6 run k6-tests/Cloud/upload-mixed/TC01-concurrent-6-users.js
k6 run k6-tests/Cloud/upload-mixed/TC02-concurrent-100-users.js
```

`lib/core-mixed.js` reads `data/upload-manifest-mixed.json` and uses the same generous
timeout/thresholds as `core-video.js` — a safe superset, since small image uploads finish
well within those limits and only the video-sized entries actually need them.

## Verify after test

### Method 1 — Download the file via URL (fast, visual)

```
https://node8.17qpb.com:3065/store/file/<file_hash>
```

Get `file_hash` from `data/upload-manifest-image.json` (field `init_body.file_hash`), then open the URL in a browser or use curl:

```powershell
curl -k -s -o /dev/null -w "HTTP Status: %{http_code}`nContent-Type: %{content_type}`nSize: %{size_download} bytes`n" "https://node8.17qpb.com:3065/store/file/<file_hash>"
```

| Result | Meaning |
|---------|---------|
| `200` + image downloads | File uploaded successfully |
| `404` | File does not exist on server |

> Note: use `node8`, not `node1` (the node used for uploading). The file is stored on a different node than the one receiving the upload request.

### Method 2 — Call POST init again

If the file was already uploaded, the server returns `Status=1`:
```powershell
cd D:\TBChatPerformance\genfile\upload
$body = Get-Content ".\index\<unique>" -Raw
Invoke-RestMethod -Method POST -Uri "https://node1.17qpb.com:3065/api/v2/upload/init" -ContentType "application/json" -Body $body
```
`data.Status = 1` → file already exists on the server (upload succeeded).
`data.Status = 2` → file does not exist → upload did not succeed.

## Reading the results

| Metric | Meaning |
|--------|---------|
| `files_completed` | Number of files uploaded successfully |
| `http_req_failed` | Request failure rate (target: 0%) |
| `init_duration p(95)` | Time within which 95% of POST init requests complete |
| `slice_upload_duration p(95)` | Time within which 95% of PUT slices complete |
| `file_upload_duration p(95)` | Time within which 95% of full-file uploads complete |
| `iteration_duration avg` | Average time to fully upload one image |

## Notes

- Test images/videos must be unique (different hashes) to avoid the server dedup skipping the upload.
- To change the server: edit `baseUrl` in `k6-tests/Cloud/config/enviroment.js`, do not edit `core-image.js`/`core-video.js` directly.
- `uploadTool.exe` only performs the PUT step, it does not replace POST init — not used in k6.
- `genfile/upload/index` and `genfile/upload/upload` are rebuilt from scratch by `buildFileTool.exe` every run — running it for videos overwrites whatever was previously built for images, and vice versa. Re-run `buildFileTool.exe` + the manifest step for the dataset you want to test right before running k6.
